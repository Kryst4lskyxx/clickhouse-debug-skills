# Source map — confirming the cause against the matched tree

The Confirm stage. Prometheus and `system.*` tell you *what* broke and *when*; only
the source tells you the version-specific *why* without guessing. This is the
skill's differentiator — diagnosis depth is delegated to `altinity-expert-*` and
remedies to `clickhouse-best-practices`, but **nobody else confirms the mechanism in
the running version's code.** Do it on every root-cause claim.

Line numbers move between releases, so this playbook gives **grep patterns and file
locations, not line numbers**. Run them in the checked-out tree.

## Step 0: confirm the tree matches the cluster (do this first)

A throw site read from the wrong version is worse than no source at all — the logic
may have moved or changed. Confirm before you cite anything:

```bash
git describe --tags                       # what you're standing in
grep -E 'VERSION_(MAJOR|MINOR|PATCH|DESCRIBE)' cmake/autogenerated_versions.txt
```

Compare to the cluster: `./chq.sh "SELECT version()"`. If they diverge, check out the
matching tag (`git checkout v24.3.5.46-lts`) before reading source, or say in the
writeup that line/behavior may differ. For a proxy-fronted fleet, versions can vary
per node — confirm the spread (`SELECT hostName(), version() FROM clusterAllReplicas(...)`)
and match the tree to the *node that failed*.

## Error code → throw site

The highest-value move: an error name from `system.errors` or a log → the exact
precondition that raises it.

```bash
# 1. name <-> number, and confirm the code exists in THIS version
grep -n "CANNOT_SCHEDULE_TASK" src/Common/ErrorCodes.cpp
# 2. every throw site (the precondition lives in the surrounding lines)
grep -rn "ErrorCodes::CANNOT_SCHEDULE_TASK" src/
```

`src/Common/ErrorCodes.cpp` defines every code via an `M(number, NAME)` macro list;
that's the code↔number map. Each throw site does
`throw Exception(ErrorCodes::NAME, "message {}", arg)` — read the `if` above it to
learn what condition the server was actually in. A code may have several throw
sites; match the one whose message string matches your log line. To find by message
instead, grep the format string: `grep -rn "Cannot reschedule" src/`.

## Metric / event → what actually increments it

A metric only means what its increment site says it means. Don't infer from the name.

```bash
# ProfileEvents (cumulative counters: query_log.ProfileEvents, system.events)
grep -n "ContextLock" src/Common/ProfileEvents.cpp                       # definition + description
grep -rn "ProfileEvents::increment(ProfileEvents::ContextLock" src/      # where/why it ticks

# CurrentMetrics (gauges: system.metrics, metric_log.CurrentMetric_*)
grep -n "BackgroundMergesAndMutationsPoolTask" src/Common/CurrentMetrics.cpp
grep -rn "CurrentMetrics::Increment.*Query\b" src/

# AsynchronousMetrics (system.asynchronous_metrics: OSMemoryTotal, MemoryResident, …)
grep -rn "OSMemoryTotal\|MemoryResident" src/Common/AsynchronousMetrics.cpp \
        src/Interpreters/ServerAsynchronousMetrics.cpp
```

Example payoff: `ProfileEvents::ContextLock` ticking hard with ~0 `ContextLockWaitMicroseconds`
is a CPU spin re-acquiring the lock, **not** lock contention — the increment site
shows the counter bumps on every acquire, waited-on or not.

## Setting → definition, default, and scope

When a fix is "set X" or a symptom is "X is 0", confirm the setting exists in this
version and what its default/scope is:

```bash
grep -n "max_concurrent_queries" src/Core/ServerSettings.cpp   # server-level (config.xml)
grep -n "max_memory_usage"       src/Core/Settings.cpp         # query/session/profile level
```

Each entry is `M(Type, name, default, "description", flags)`. **Version caveat:** older
trees keep these in `src/Core/Settings.h` / `ServerSettings.h` (macro-based) rather
than `.cpp`; if the `.cpp` grep is empty, retry the `.h`. The distinction matters for
the writeup: `max_memory_usage` is per-query (a profile/session knob), while
`max_server_memory_usage` and `max_concurrent_queries` are server settings that need
a config change + reload.

## system.* column → meaning

When you're unsure what a `system.*` column reports (and the docs are ambiguous for
this version), the table's storage class is authoritative:

```bash
ls src/Storages/System/ | grep -i errors          # -> StorageSystemErrors.cpp
grep -rn "absolute_delay" src/Storages/System/StorageSystemReplicas.cpp
```

`src/Storages/System/StorageSystem<Name>.cpp` builds each row — it shows exactly how
a column is computed (e.g. how `system.replicas.absolute_delay` is derived, or what
`is_readonly` actually tracks).

## Behavior / crash → mechanism

For "why did a symptom become a *crash*", trace the mechanism, not just a code:

- **Pool sizing / scheduling** → `src/Common/ThreadPool.cpp`. `CANNOT_SCHEDULE_TASK
  "failed to start the thread"` comes from a failed `std::thread` ctor (the OS
  refusing `clone()`), not a quota — which is why the lever is `max_concurrent_queries`,
  not `max_thread_pool_size`.
- **Throwing destructors / unwinding** → an exception escaping a destructor during
  stack unwinding hits `std::terminate` → SIGSEGV. Grep the relevant destructor
  (e.g. hedged-connections teardown) to confirm the throw path, which is why
  `use_hedged_requests=0` converts the crash into a clean per-query error.
- **Signal handling** → `src/Daemon/BaseDaemon.cpp` (signal handler, fault address,
  the stack the server logs on crash).
- **Memory tracker** → `src/Common/MemoryTracker.cpp` — the `MEMORY_LIMIT_EXCEEDED`
  message, and why untracked allocations (cross-joins) can overshoot RSS past the
  tracked figure shown in `query_log`.

## Worked examples (codes this skill already names)

- **`CANNOT_SCHEDULE_TASK`** → `grep -rn "ErrorCodes::CANNOT_SCHEDULE_TASK" src/Common/ThreadPool.cpp`
  — the catch around the `std::thread` ctor. Confirms an admission stampede (OS
  refusing threads), not pool saturation.
- **`TOO_MANY_PARTS`** → `grep -rn "ErrorCodes::TOO_MANY_PARTS" src/Storages/MergeTree/`
  — the `parts_to_throw_insert` check in `MergeTreeData`. Confirms over-insertion vs.
  a *failing-merge storm* (parts pinned at the limit while merges error at ~0ms).
- **`MEMORY_LIMIT_EXCEEDED`** → `grep -rn "ErrorCodes::MEMORY_LIMIT_EXCEEDED" src/Common/MemoryTracker.cpp`
  — which limit (query / user / server) and the message format, so you cite the right cap.
- **`FILE_DOESNT_EXIST`** during merges → `grep -rn "ErrorCodes::FILE_DOESNT_EXIST" src/`
  + read the merge path; confirms a dead/corrupt disk under the storm rather than
  application error.

Cite the `file:line` you land on (from the matched tree) in the RCA writeup. If the
source contradicts your hypothesis, believe the source.
